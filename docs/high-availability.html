High Availability

Run multiple Sertone gateways to eliminate single points of failure. When one gateway goes offline — for maintenance, updates, or an unexpected outage — the others keep your services running without interruption.

Overview

A single Sertone gateway is already highly reliable — it runs as a self-contained Docker container with automatic recovery. But for operators who need guaranteed uptime or need to apply updates without any service interruption, running multiple gateways in a cluster is the recommended setup.

Gateways in an HA cluster:

  • Share load — consumers can connect to any gateway in the cluster and receive the same services.
  • Detect each other — gateways communicate over the Sertone Global Network itself. No additional infrastructure is needed.
  • Update live — you can roll out a new version to one gateway while others continue serving. Once the updated gateway is healthy, the next one updates.
  • Survive individual failures — if one gateway loses power, crashes, or loses its internet connection, consumers reroute to another within seconds.

How It Works

Each gateway in an HA cluster has its own identity on the network — a unique address that consumers and peers can reach it by. All gateways in a cluster register the same services and share the same catalog. When a consumer calls a service, any gateway in the cluster can handle the request.

Gateways automatically discover each other through the mesh network and maintain connections between themselves. This connection is used to:

  • Synchronize service registration and catalog metadata
  • Forward incoming requests when a specific service is available on a peer but not locally
  • Signal availability — peers mark each other as healthy or unreachable based on live connection state
No shared database required. Each gateway maintains its own local state. There is no distributed database, no consensus cluster, no Zookeeper. Coordination happens over the mesh using the same encrypted protocol used for API calls.

Deployment Setup

To run an HA cluster, deploy two or more Sertone gateways on different machines (or cloud regions). Each gateway is a standard Docker container configured with the same service catalog settings.

Minimum setup: 2 gateways

Two gateways give you basic redundancy — if one goes down, the other takes over within seconds. This is sufficient for most use cases.

Recommended setup: 3 or more gateways

Three gateways allow one to be taken offline for maintenance while the other two continue serving. This is the recommended setup for production workloads with SLA requirements.

Geographic distribution

For global services, distributing gateways across regions reduces latency for consumers in different parts of the world. The Sertone routing layer automatically selects the nearest available gateway.

Zero-Downtime Updates

Sertone gateways use automated container updates — when a new version is available, each gateway updates itself automatically. In an HA cluster, gateways stagger their updates so at least one is always available during the update window.

The update process for a single gateway in an HA cluster:

  1. Gateway detects a new version is available.
  2. It signals to its peers that it is entering maintenance mode.
  3. Peers stop routing new requests to it; in-flight requests complete normally.
  4. Gateway updates and restarts — typically takes under 60 seconds.
  5. Gateway re-joins the cluster and resumes serving requests.
  6. Peers restore it to the active rotation.

Monitoring an HA Cluster

Your gateway's dashboard shows the cluster status, including how many peers are currently active and their relative health. The Mesh section of the dashboard displays:

  • Active peers — the number of other gateways currently connected
  • Inbound connections — other network participants connected to this gateway
  • Network health — gossip round-trip success rate and latency per peer

If a peer goes offline, the dashboard reflects this within the gossip cycle (typically under 2 minutes). Email alerts can be configured in Settings → Alerts to notify you when a peer becomes unreachable.

Failover Behavior for Consumers

Consumers connecting to your service do not need to configure anything for failover to work. The Sertone Global Network handles routing automatically. When a consumer calls a service:

  1. The network identifies all gateways hosting that service.
  2. The request is routed to the nearest reachable gateway.
  3. If that gateway becomes unreachable mid-request, the consumer's gateway retries via a different peer.

From the consumer's perspective, a failover event is transparent — the API call succeeds, possibly with slightly higher latency during the switchover.

Frequently Asked Questions

Do all gateways in an HA cluster need to be in the same region?

No. Gateways communicate over the global Sertone network regardless of physical location. In fact, distributing across regions is recommended for better global latency and resilience against regional outages.

Can I add or remove gateways from a running cluster?

Yes. Adding a new gateway is as simple as deploying it — it discovers existing cluster members automatically through the mesh. Removing a gateway: shut it down, and remaining peers detect the absence within minutes and adjust routing accordingly.

What happens to in-flight API calls during a failover?

Calls that are in progress when a gateway goes down will return an error to the consumer. The consumer SDK retries automatically (configurable — see the SDK documentation). If a request fails, the consumer is not charged.

Is there a limit to how many gateways I can run in a cluster?

There is no hard limit. The Sertone mesh protocol is designed to scale to thousands of participants globally. Most operators run between 2 and 10 gateways in a cluster.

Can different gateways in a cluster offer different services?

Yes. Each gateway manages its own service catalog independently. You can have some gateways offer the full catalog and others offer a subset, or specialize different gateways for different service types. The network routes consumers to a gateway that offers the service they requested.
