High Availability

Zero Downtime. Any Scale. No Complexity.

Run multiple Sertone gateways as a cluster. They detect each other automatically, share the load, update one at a time, and keep your services running no matter what happens to any individual gateway.

Zero-Downtime Updates — How It Works

When a new version is available, gateways update themselves one at a time. Here is the exact sequence for a single gateway in a 3-gateway cluster:

  1. Gateway signals peers — announces it is entering maintenance
  2. Peers stop sending new requests to it — active requests complete normally
  3. Gateway pulls new container image — typically completes in under 60 seconds
  4. Gateway restarts and rejoins — passes health checks, announces it is ready
  5. Peers restore it to the active rotation — back to full capacity

Throughout this process, the other two gateways continue serving without any interruption. From the consumer's perspective, the update is invisible.

Setting Up an HA Cluster

Running a cluster is no different from running a single gateway — just deploy multiple containers. They find each other automatically.

# Run 3 gateways on a single machine (different ports)
# For production: spread across different physical servers
services:
  sertone-1:
    image: sertonenet/gateway:latest
    ports:
      - "80:80"
      - "3002:3002"
    volumes:
      - sertone-data-1:/app/data
    restart: unless-stopped

  sertone-2:
    image: sertonenet/gateway:latest
    ports:
      - "8081:80"
      - "3003:3002"
    volumes:
      - sertone-data-2:/app/data
    restart: unless-stopped

  sertone-3:
    image: sertonenet/gateway:latest
    ports:
      - "8082:80"
      - "3004:3002"
    volumes:
      - sertone-data-3:/app/data
    restart: unless-stopped

volumes:
  sertone-data-1:
  sertone-data-2:
  sertone-data-3:
# Server A (New York) — identical command on each server
server-a$ docker run -d \
  --name sertone \
  -p 80:80 -p 3002:3002 \
  -v sertone-data:/app/data \
  --restart unless-stopped \
  sertonenet/gateway:latest

# Server B (London) — same image, different data volume
server-b$ docker run -d \
  --name sertone \
  -p 80:80 -p 3002:3002 \
  -v sertone-data:/app/data \
  --restart unless-stopped \
  sertonenet/gateway:latest

# That's it. Gateways find each other through the Sertone network.
# No configuration file needed. No shared database. No orchestrator.

# Check cluster status from either gateway
server-a$ curl http://localhost/api/mesh/status
{
  "neighbors": { "count": 2, "target": 3 },
  "gossip": { "london-gateway": { "ok": 847, "fail": 0, "lastOkSec": 12 } }
}
# Poll cluster health — check from any gateway in the cluster
$ watch -n 30 'curl -s http://localhost/api/mesh/status | jq "{neighbors, gossip}"'

# Response shows live peer health
{
  "neighbors": {
    "count": 2,
    "target": 3,
    "ids": ["lon-gw", "sgp-gw"]
  },
  "gossip": {
    "lon-gw": { "ok": 1203, "fail": 0, "lastOkSec": 8 },
    "sgp-gw": { "ok": 1198, "fail": 2, "lastOkSec": 21 }
  }
}

# "fail" counter non-zero = peer had connectivity issues
# "lastOkSec" > 120 = peer may be offline

Start Your HA Cluster

1

Deploy Your First Gateway

Pull and run the free Sertone Docker container on any machine. Register your services through the web console at https://your-server:3002/panel.

docker run -d --name sertone \
  -p 80:80 -p 3002:3002 \
  -v sertone-data:/app/data \
  sertonenet/gateway:latest
2

Add More Gateways

Run the same container on additional machines — different cloud provider, different region, whatever you prefer. Each gateway discovers the others automatically through the Sertone network within minutes of starting.

3

Verify the Cluster

Open the Mesh section in your gateway's dashboard. You will see all active peers, their gossip health, and connection latency. No additional configuration required.
