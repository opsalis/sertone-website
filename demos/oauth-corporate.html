Corporate API Access with SSO

Protect your services with your company's existing identity provider — Keycloak, Okta, Azure AD, or Auth0. Employees log in once with corporate credentials. No separate accounts, no API keys to distribute, no manual onboarding.

OAuth 2.0 OpenID Connect Keycloak JWT Validation Webcam / Video Enterprise
The Problem

Companies already have an identity provider

ACME Corporation runs 40 security cameras across their offices. They want employees to watch feeds from anywhere, but only employees. The naive approach — API keys, shared passwords, or a custom auth service — creates real problems:

API Keys Break

Distributed API keys get shared, rotated poorly, and leak. When an employee leaves, you can't easily revoke all their access.

Separate Accounts

Asking employees to create yet another account causes low adoption and password reuse. IT has to manage two identity systems.

Manual Onboarding

Granting and revoking access one employee at a time does not scale. When a new hire joins, waiting a week for access defeats the purpose.

The Sertone solution: configure your Sertone node once with your company's OAuth server. Every employee who can log in to their corporate account automatically gets access to the services you assign them. When someone leaves and IT disables their corporate account, their access to the stream is revoked the next time the token is checked. No separate management needed.

The Sertone Global Network uses a proprietary fully encrypted protocol. All traffic is end-to-end encrypted. Payment is automatic.

Through its built-in web console, you browse services, register your own, manage your wallet, and monitor your earnings. Installation takes minutes. It is completely free, forever.

docker run -d --name my-sertone -p 3000:3000 -p 3002:3002 sertone/wrapper:latest

Request trace — step by step

Steps 1–3

Employee opens stream URL. The Sertone node checks for a valid session. None found. Node returns 302 Found — redirect to the Keycloak login page in Paris.

Steps 4–6

Employee logs in with corporate credentials. Keycloak validates against the company directory (LDAP/AD). On success, Keycloak issues an authorization code and redirects back to the Sertone node.

Steps 7–9

Sertone exchanges the code for tokens. This is a server-to-server call — the client secret never leaves the node. The JWT is validated: signature, expiry, audience, issuer. Session created.

Step 10

Webcam stream delivered. Media is relayed through Tokyo for bandwidth optimization. The employee's browser receives the stream. Usage is billed to the ACME Corp subscription account.

How It Works

Step-by-step configuration

Four steps from nothing to a fully protected corporate stream. Each step maps directly to what you'd do in production.

Step A — Company admin configures the Sertone node

A1
Sertone Node — New York

Open Settings > Corporate Customers

In the Sertone web console (https://localhost:3002/panel), navigate to Settings in the sidebar, then select the Corporate Customers tab. Click "Add Company".

A2
Sertone Node — New York

Enter company details and OAuth configuration

Fill in the company name (ACME Corp), the corporate email domain (acme-corp.example), and the OAuth provider type. Then provide the OpenID Connect configuration:

# Fields you enter in the Sertone web console Company name: ACME Corp Email domain: acme-corp.example Provider: Keycloak Issuer URL: https://auth.acme-corp.example/realms/acme Client ID: sertone-gateway Client Secret: [your-secret-from-keycloak] Auth endpoint: https://auth.acme-corp.example/realms/acme/protocol/openid-connect/auth Token endpoint: https://auth.acme-corp.example/realms/acme/protocol/openid-connect/token JWKS URI: https://auth.acme-corp.example/realms/acme/protocol/openid-connect/certs Scopes: openid email profile groups
A3
Sertone Node — New York

Select which services are available to ACME employees

In the "Services" tab of the ACME Corp company record, toggle on the streams and APIs that employees should access. For this demo: "Security Camera — Lobby", "Security Camera — Parking", "Security Camera — Server Room".

Step B — Configure Keycloak (self-hosted, free)

B1
Keycloak — Paris

Start Keycloak with Docker

ACME runs their own Keycloak instance. A single Docker command is enough to get started. For production, Keycloak should be behind a reverse proxy with TLS.

# Start Keycloak in dev mode (replace with production config for live use) $ docker run -d \ --name keycloak \ -p 8080:8080 \ -e KEYCLOAK_ADMIN=admin \ -e KEYCLOAK_ADMIN_PASSWORD=change-me-now \ quay.io/keycloak/keycloak:25.0 start-dev # Keycloak is now running at http://localhost:8080 # Admin console: http://localhost:8080/admin
B2
Keycloak — Paris

Create the "acme" realm

A realm is an isolated tenant in Keycloak. ACME gets their own realm so their users are completely separate from any other company using the same Keycloak instance. Get an admin token first, then create the realm.

# Get admin access token $ ADMIN_TOKEN=$(curl -s -X POST \ http://localhost:8080/realms/master/protocol/openid-connect/token \ -d "client_id=admin-cli&grant_type=password&username=admin&password=change-me-now" \ | jq -r .access_token) # Create the "acme" realm $ curl -s -X POST http://localhost:8080/admin/realms \ -H "Authorization: Bearer $ADMIN_TOKEN" \ -H "Content-Type: application/json" \ -d '{ "realm": "acme", "displayName": "ACME Corporation", "enabled": true, "sslRequired": "external", "registrationAllowed": false, "loginWithEmailAllowed": true, "accessTokenLifespan": 3600 }'
B3
Keycloak — Paris

Create an OAuth client for the Sertone node

The Sertone node needs a confidential client registered in Keycloak. This client uses the Authorization Code flow — the most secure option for server-side applications. The redirect URI must point back to your Sertone node's OAuth callback.

# Create a confidential OAuth2 client for Sertone $ curl -s -X POST \ http://localhost:8080/admin/realms/acme/clients \ -H "Authorization: Bearer $ADMIN_TOKEN" \ -H "Content-Type: application/json" \ -d '{ "clientId": "sertone-gateway", "name": "Sertone Node Gateway", "enabled": true, "publicClient": false, "secret": "your-secret-here-change-me", "standardFlowEnabled": true, "directAccessGrantsEnabled": false, "redirectUris": [ "https://localhost:3006/auth/callback", "https://your-node.acme-corp.example:3006/auth/callback" ], "webOrigins": [ "https://localhost:3006", "https://your-node.acme-corp.example:3006" ], "protocol": "openid-connect", "attributes": { "access.token.lifespan": "3600" } }' # The client secret is what you enter in Sertone web console # Rotate it with: POST /admin/realms/acme/clients/{id}/client-secret
B4
Keycloak — Paris

Create users (or connect to Active Directory / LDAP)

For small deployments, create users directly in Keycloak. For enterprise environments, connect Keycloak to the existing Active Directory or LDAP server — Keycloak calls it a "User Federation". Existing corporate users log in immediately, no migration needed.

# Create user: alice $ curl -s -X POST \ http://localhost:8080/admin/realms/acme/users \ -H "Authorization: Bearer $ADMIN_TOKEN" \ -H "Content-Type: application/json" \ -d '{ "username": "alice", "email": "alice@acme-corp.example", "firstName": "Alice", "lastName": "Durand", "enabled": true, "emailVerified": true, "credentials": [{ "type": "password", "value": "demo2026", "temporary": false }], "groups": ["/employees", "/security-ops"] }' # Create user: bob (read-only, no security-ops group) $ curl -s -X POST \ http://localhost:8080/admin/realms/acme/users \ -H "Authorization: Bearer $ADMIN_TOKEN" \ -H "Content-Type: application/json" \ -d '{ "username": "bob", "email": "bob@acme-corp.example", "firstName": "Bob", "lastName": "Martin", "enabled": true, "emailVerified": true, "credentials": [{ "type": "password", "value": "demo2026", "temporary": false }], "groups": ["/employees"] }' # To connect to Active Directory instead of managing users manually: # Keycloak admin console > User Federation > Add LDAP provider # Enter your AD host, bind DN, users DN, and sync settings.
B5
Keycloak — Paris

Add a groups claim to the access token

By default, Keycloak does not include group memberships in the JWT. Add a mapper so the Sertone node can check which groups the employee belongs to and grant the right level of access.

# Get the client's internal ID $ CLIENT_ID=$(curl -s \ http://localhost:8080/admin/realms/acme/clients?clientId=sertone-gateway \ -H "Authorization: Bearer $ADMIN_TOKEN" \ | jq -r '.[0].id') # Add a "groups" protocol mapper to include group names in the JWT $ curl -s -X POST \ http://localhost:8080/admin/realms/acme/clients/$CLIENT_ID/protocol-mappers/models \ -H "Authorization: Bearer $ADMIN_TOKEN" \ -H "Content-Type: application/json" \ -d '{ "name": "groups", "protocol": "openid-connect", "protocolMapper": "oidc-group-membership-mapper", "config": { "claim.name": "groups", "full.path": "false", "id.token.claim": "true", "access.token.claim": "true", "userinfo.token.claim": "true" } }'

Step C — Employee accesses the webcam stream

C1
Employee Browser

Employee opens the stream URL

The employee navigates to the stream URL on the company's Sertone node. No special client software needed — a standard browser is enough.

https://your-node.acme-corp.example:3006/stream/security-cam-lobby
C2
Sertone Node — New York

Sertone redirects to corporate login

The node detects no valid session cookie. It generates a state parameter (CSRF protection) and a nonce (replay protection), stores them in a temporary session, then returns a redirect to Keycloak with all required OAuth parameters.

# What the Sertone node sends to the browser: HTTP/1.1 302 Found Location: https://auth.acme-corp.example/realms/acme/protocol/openid-connect/auth ?response_type=code &client_id=sertone-gateway &redirect_uri=https%3A%2F%2Fyour-node.acme-corp.example%3A3006%2Fauth%2Fcallback &scope=openid+email+profile+groups &state=a9f3c2d1e8b7054f &nonce=7f4e2b9c1a3d8e56
C3
Keycloak — Paris

Employee enters corporate credentials

The browser loads the Keycloak login page. This page is served entirely by ACME's own Keycloak server — Sertone never sees the employee's password. The employee enters their regular corporate username and password.

C4
Keycloak — Paris

Keycloak validates, issues authorization code, redirects back

# Keycloak redirects back to the Sertone node with a one-time code: GET https://your-node.acme-corp.example:3006/auth/callback ?code=eyJhbGc...one-time-auth-code &state=a9f3c2d1e8b7054f
C5
Sertone Node — New York

Sertone exchanges code for tokens (server-to-server)

The node validates the state parameter (CSRF check), then makes a back-channel POST to Keycloak's token endpoint. The client secret is included — it never passes through the employee's browser.

# Server-to-server token exchange — happens inside the Sertone node POST https://auth.acme-corp.example/realms/acme/protocol/openid-connect/token Content-Type: application/x-www-form-urlencoded grant_type=authorization_code &code=eyJhbGc...one-time-auth-code &redirect_uri=https://your-node.acme-corp.example:3006/auth/callback &client_id=sertone-gateway &client_secret=your-secret-here-change-me # Keycloak responds with: { "access_token": "eyJhbGciOiJSUzI1NiIsInR5...", "id_token": "eyJhbGciOiJSUzI1NiIsInR5...", "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5...", "token_type": "Bearer", "expires_in": 3600 }
C6
Sertone Node — New York

JWT validation — five checks

The Sertone node validates the JWT before granting access. All five checks must pass:

# What Sertone checks on every token 1. Signature: Verified against Keycloak's JWKS endpoint (public keys) GET https://auth.acme-corp.example/realms/acme/protocol/openid-connect/certs 2. Expiry: Token must not be expired (exp claim) token.exp > Date.now() / 1000 3. Issuer: Must match configured issuer token.iss === "https://auth.acme-corp.example/realms/acme" 4. Audience: Must include our client ID token.aud.includes("sertone-gateway") 5. Nonce: Must match the nonce sent in the auth request (replay protection) token.nonce === stored_nonce # On success: session cookie issued, employee is redirected to the original URL # On failure: 401 — redirect back to login
C7
Sertone Node — New York → Tokyo

Webcam stream delivered

Token valid, session created. The employee is redirected to the original stream URL. This time the session cookie is present — the stream is proxied through the Tokyo media relay node for bandwidth optimization, and plays in the browser. Usage is billed to the ACME Corp subscription at the configured rate.

For Consumers

How to use this service

Any authorized employee or developer can access a corporate-protected service through their own Sertone control center. Here is the complete flow from installation to first authenticated call:

1

Install your Sertone control center

One Docker command on any machine -- your laptop, a Raspberry Pi, a cloud VM.

$ docker run -d --name my-sertone -p 3000:3000 -p 3002:3002 sertone/wrapper:latest
2

Browse the catalog

Open your web console at https://localhost:3002/panel. Go to the Catalog tab. Search for the service you need -- use keywords, tags, or browse by category.

3

Authenticate with your corporate identity provider

When you first access an OAuth-protected service, your Sertone redirects you to your company's login page (Keycloak, Okta, Azure AD, or Auth0). Log in once with your corporate credentials. Your Sertone stores the session token and renews it automatically -- you will not be prompted again on subsequent calls.

4

Fund your wallet

Go to the Finance tab. Deposit USDC and a small amount of ETH for gas. Your control center shows both balances and alerts you when they are low.

5

Call it from your code

Copy your consumer secret from Settings. Use the service UUID from the catalog. Your Sertone attaches the corporate JWT automatically on every call:

$ curl -X POST https://localhost:3000/internal/call \ -H "Authorization: Bearer YOUR_CONSUMER_SECRET" \ -H "Content-Type: application/json" \ -d '{"api_id_public": "SERVICE_UUID", "path": "/stream/security-cam-lobby", "method": "GET"}'

Or generate an SDK in your preferred language (Python, TypeScript, Go, Rust, Kotlin, and more) -- the web console generates ready-to-use client code with the UUID pre-filled.

6

You are in business

Every call is paid automatically in USDC. You pay the service owner's price. No invoices, no payment terms, no chargebacks. Your Finance tab shows every transaction.

Replicate This on Your Raspberry Pi

Run a corporate-grade authenticated webcam stream from a Raspberry Pi with a USB camera attached. Five steps from unboxing to a secured stream.

1

Install Docker and start the Sertone node

Run the Sertone container. It starts on ports 3000 (Sertone Global Network), 3002 (web console), and 3006 (subscription / streaming).

$ curl -fsSL https://get.docker.com | sh $ docker run -d \ --name sertone \ -p 3000:3000 -p 3002:3002 -p 3006:3006 \ -v sertone-data:/data \ sertone/wrapper:latest
2

Install Keycloak and run the full setup script

Keycloak can run on the same Raspberry Pi or a separate machine. For production, run it on a machine that is always on and behind a proper TLS reverse proxy (nginx or Caddy).

$ docker run -d \ --name keycloak \ -p 8080:8080 \ -e KEYCLOAK_ADMIN=admin \ -e KEYCLOAK_ADMIN_PASSWORD=change-me-now \ quay.io/keycloak/keycloak:25.0 start-dev # Wait ~30 seconds for Keycloak to start, then run the setup script # (see the "Keycloak admin API" tab in Code Samples above)
3

Stream your webcam with FFmpeg

FFmpeg reads the USB webcam and streams it in MJPEG format on a local port. The Sertone node proxies this stream and adds authentication on top.

# Install ffmpeg $ sudo apt-get install -y ffmpeg # Stream USB webcam (/dev/video0) as MJPEG on port 8554 $ ffmpeg -f v4l2 -i /dev/video0 \ -vf scale=1280:720 \ -q:v 5 \ -f mjpeg \ -listen 1 \ http://0.0.0.0:8554/stream # The stream is now accessible at http://localhost:8554/stream # Not exposed to the internet — Sertone node proxies it with auth
4

Configure Corporate Customer in Sertone web console

Open the Sertone web console at https://localhost:3002/panel. Go to Settings → Corporate Customers → Add Company. Fill in the Keycloak details from step 2.

Company name: ACME Corp Email domain: acme-corp.example Provider type: Keycloak / OpenID Connect Issuer URL: http://localhost:8080/realms/acme Client ID: sertone-gateway Client Secret: your-secret-here-change-me Redirect URI: https://localhost:3006/auth/callback
5

Register the webcam stream as a service

In the Sertone web console, go to APIs → Register New. Set the upstream to the FFmpeg stream and assign it to the ACME Corp corporate customer.

Name: Security Camera — Lobby Upstream URL: http://localhost:8554/stream Protocol: MJPEG / HTTP stream Auth: Corporate SSO — ACME Corp Price: $0.002 / minute Groups: security-ops (only employees in this group can view)
6

Test: open the stream — get redirected — log in — watch

Open a browser and navigate to the stream URL. You should be redirected to the Keycloak login page. Log in with alice / demo2026. The webcam stream plays immediately after login.

# Stream URL (opens in browser — redirects to Keycloak login): https://localhost:3006/stream/security-cam-lobby # If you want to test with curl (non-browser, password flow): $ TOKEN=$(curl -s -X POST \ http://localhost:8080/realms/acme/protocol/openid-connect/token \ -d "grant_type=password&username=alice&password=demo2026" \ -d "client_id=sertone-gateway&client_secret=your-secret-here-change-me" \ | jq -r .access_token) $ curl -I "https://localhost:3006/stream/security-cam-lobby" \ -H "Authorization: Bearer $TOKEN" # Expected: HTTP/2 200 — stream starts
Supported Providers

Any OpenID Connect provider works

Sertone uses the standard OpenID Connect / OAuth 2.0 Authorization Code flow. Any provider that implements the OIDC standard is compatible — no custom integration required.

Provider Type Cost Self-hosted LDAP / AD integration Groups in JWT
Keycloak Open source, full-featured IdP Free Yes Built-in Via mapper
Okta Cloud identity platform From $2/user/month No Yes Native
Azure AD Microsoft cloud identity Included in M365 No Native AD Via optional claim
Auth0 Cloud identity platform Free up to 7,500 MAU No Enterprise plan Via action/rule
Google Workspace Google cloud identity Included in Workspace No Via GCDS Via Admin SDK
Authentik Open source IdP Free Yes Built-in Native

How Sertone discovers provider configuration: When you enter the issuer URL, Sertone fetches {issuer}/.well-known/openid-configuration and reads the token endpoint, auth endpoint, and JWKS URI automatically. You only need to provide the issuer URL, client ID, and client secret.
